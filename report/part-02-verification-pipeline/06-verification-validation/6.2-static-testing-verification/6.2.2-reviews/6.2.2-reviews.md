### 6.2.2. Reviews

Las revisiones de código son un proceso fundamental en el desarrollo de StockSip para garantizar la calidad y la conformidad con las normas establecidas en todos los repositorios de la organización LiquoTrack-EXP. Este proceso implica tanto revisiones manuales como automáticas y sigue las siguientes pautas:

**Tipos de Revisiones:**

- **Revisión de Código por Pares:** Un integrante del equipo de StockSip revisa el código de otro desarrollador para garantizar que cumpla con los estándares definidos, como las convenciones de Conventional Commits, los principios de Clean Code y la correcta aplicación del Ubiquitous Language del dominio de licorerías. Esta práctica asegura que el código sea comprensible y coherente con la arquitectura del sistema.

- **Revisión de Código Formal:** Incluye una reunión estructurada al cierre de cada sprint donde el equipo evalúa el código desarrollado utilizando un checklist, lo que permite detectar problemas de manera grupal antes de integrar los cambios a la rama principal. Esta revisión es especialmente importante para los bounded contexts críticos como Inventory Management y Order Management.

- **Revisión Automática:** El equipo utiliza SonarLint durante el desarrollo en tiempo real y SonarQube para el análisis continuo de los repositorios de StockSip, detectando errores, vulnerabilidades de seguridad y problemas de calidad en el código del backend en ASP.NET Core, la aplicación web en Vue y la aplicación móvil en Kotlin.

**Proceso de Revisión:**

- **Creación de Pull Requests:** Los desarrolladores de StockSip deben crear un Pull Request desde su rama de tipo feature o bugfix hacia la rama develop, incluyendo una descripción clara de los cambios realizados, las historias de usuario asociadas y las pruebas implementadas para validar el comportamiento del código modificado.

- **Checklist de Revisión:** Debe existir una lista de verificación que cubra aspectos como la claridad del código, el uso correcto del lenguaje ubicuo del dominio, la cobertura de pruebas unitarias e integración, el manejo adecuado de errores en los endpoints de la API RESTful y la ausencia de vulnerabilidades de seguridad en las operaciones de gestión de inventario y autenticación de usuarios.

- **Comentarios y Feedback:** Los revisores deben proporcionar comentarios constructivos y específicos sobre el código evaluado. Cualquier problema identificado, ya sea en la lógica de negocio de los bounded contexts, en la estructura de los endpoints o en la interfaz de usuario, debe ser abordado y corregido por el autor del PR antes de proceder con la aprobación final.

- **Aprobación o Rechazo de PR:** El Pull Request debe ser aprobado por al menos un integrante adicional del equipo antes de fusionarse en la rama develop. Solo los cambios verificados y aprobados podrán avanzar posteriormente hacia la rama release para su despliegue en los entornos de producción alojados en Vercel y Azure.

**Criterios de Aceptación:**

- **Calidad y Seguridad del Código:** El código debe cumplir con los estándares de calidad definidos por el equipo de StockSip y no debe introducir vulnerabilidades de seguridad, especialmente en operaciones sensibles como el registro de usuarios, la gestión de credenciales de empleados y el procesamiento de pagos a través de Mercado Pago.

- **Cobertura de Pruebas:** Se requiere una cobertura mínima del 80% en pruebas unitarias e integración para asegurar que el código nuevo esté adecuadamente validado, en línea con las suites de testing implementadas para los bounded contexts del sistema.

**Frecuencia de Revisiones:** El equipo de StockSip realiza revisiones de código de forma regular al finalizar cada sprint, asegurando que los cambios no se acumulen y que la calidad del código se mantenga consistente en todos los productos de software del proyecto, incluyendo la landing page, la aplicación web, el backend y las aplicaciones móviles.
