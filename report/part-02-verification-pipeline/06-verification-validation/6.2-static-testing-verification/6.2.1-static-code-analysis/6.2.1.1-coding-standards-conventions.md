## 6.2. Static Testing & Verification

### 6.2.1. Static Code Analysis

Esta sección se centra en los métodos de prueba estática y verificación del código de StockSip, asegurando que el software cumpla con los estándares de calidad y seguridad antes de su ejecución. Estos métodos permiten identificar defectos en una fase temprana del ciclo de vida del desarrollo. El análisis de código estático implica la revisión del código fuente sin necesidad de ejecutarlo, utilizando herramientas automatizadas y revisiones manuales. Este enfoque ayuda a detectar errores, vulnerabilidades de seguridad y oportunidades de mejora en el código, lo que contribuye a aumentar la calidad general del software y a reducir el costo de las correcciones en etapas posteriores del desarrollo.

#### 6.2.1.1. Coding Standards & Code Conventions

Las normas de codificación y las convenciones son directrices que el equipo de StockSip debe seguir para garantizar un código legible, mantenible y coherente en todos los productos de software del proyecto, incluyendo la aplicación web en Vue, el backend en ASP.NET Core, la aplicación móvil nativa en Kotlin y la aplicación multiplataforma en Flutter. A continuación, se aplican los siguientes principios:

- **Clean Code:** Es fundamental utilizar nombres claros y descriptivos para variables, funciones y clases en todos los componentes de StockSip. Por ejemplo, en el backend, los nombres de los métodos deben reflejar con precisión las operaciones del dominio, como CreateWarehouse, RegisterProduct o UpdateStock. Las funciones deben ser cortas y enfocarse en una sola responsabilidad, eliminando el código muerto y los comentarios innecesarios. Este enfoque contribuye a una mayor comprensión del código y facilita la colaboración entre los miembros del equipo de desarrollo.

- **Domain-Driven Design (DDD):** El equipo emplea un lenguaje ubicuo que refleja los términos definidos en el Ubiquitous Language de StockSip, como Inventory, Warehouse, Supplier, Replenishment y Conservation Guide, tanto en el código fuente como en la comunicación entre los integrantes. El sistema se divide en bounded contexts claramente delimitados, como Inventory Management, Order Management, IAM, Payment and Subscriptions y Alerts and Notifications, utilizando entidades y objetos de valor de manera apropiada en cada contexto. La lógica del dominio es gestionada a través de servicios de dominio y repositorios, promoviendo una arquitectura organizada y alineada con los objetivos del negocio de gestión de licorerías.

- **Conventional Commits:** El equipo aplica la convención de mensajes de commit estructurados con prefijos como feat, fix, docs, style, refactor, chore y test, acompañados del scope correspondiente a la sección del sistema modificada. Esta práctica garantiza un historial de cambios claro y trazable en todos los repositorios de la organización LiquoTrack-EXP en GitHub.
