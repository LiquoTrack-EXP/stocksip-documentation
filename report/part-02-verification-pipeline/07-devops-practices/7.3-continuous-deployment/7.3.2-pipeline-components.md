### 7.3.2. Production Deployment Pipeline Components

Este apartado describe los componentes que forman parte del pipeline de despliegue a producción y cómo se integran para automatizar todo el proceso.


**Componentes del Pipeline de la Base de Datos (Azure Cosmos DB):**

Este apartado describe los componentes que forman parte del pipeline de despliegue a producción para la base de datos NoSQL gestionada en Azure Cosmos DB.

1. **Aprovisionamiento del clúster:** Al crear el recurso en el portal de Azure, se selecciona **Azure Cosmos DB for MongoDB** con el tipo de clúster **vCore**, configurando los parámetros de nombre de cuenta, grupo de recursos, ubicación y opciones de rendimiento. Este proceso genera un clúster listo para recibir conexiones desde la aplicación backend.

2. **Configuración de reglas de acceso (Firewall):** Una vez creado el clúster, se configuran las reglas de firewall en el portal de Azure para habilitar el acceso desde los entornos autorizados, incluyendo el acceso local para los miembros del equipo a través de **MongoDB Compass** y el acceso desde la aplicación desplegada en Azure Web App.

3. **Conexión desde el backend:** La aplicación backend en **ASP.NET Core con C#** se conecta a la base de datos mediante la cadena de conexión proporcionada por Azure Cosmos DB. Esta cadena se configura como variable de entorno en la sección **Configuration** del servicio de Azure Web App, garantizando que las credenciales no queden expuestas en el repositorio.

4. **Verificación de datos:** Una vez establecida la conexión, se verifica la correcta creación de colecciones y documentos desde **MongoDB Compass**, validando que las operaciones de lectura y escritura funcionen correctamente en el entorno de producción.

5. **Despliegue continuo de esquema:** Dado que MongoDB es una base de datos orientada a documentos, los cambios en el modelo de datos se reflejan automáticamente al desplegar una nueva versión del backend, sin requerir migraciones explícitas. Las nuevas colecciones y campos se crean en tiempo de ejecución según la lógica definida en la aplicación.


**Componentes del Pipeline del Backend (Azure Web App para ASP.NET Core):**

1. **Integración continua:** Al realizar un commit en la rama de desarrollo del repositorio **stocksip-platform-api** en GitHub, el equipo empuja los cambios a la rama `release/` correspondiente, desde la cual se inicia el proceso de publicación hacia Azure directamente desde el IDE **Rider** mediante el plugin **Azure Toolkit for Rider**.

2. **Publicación del proyecto:** Desde Rider, se selecciona la opción **Publish** sobre el proyecto y se elige **Azure Web App** como destino de despliegue. El IDE empaqueta la aplicación con todas sus dependencias y la envía al servicio configurado bajo el dominio **stocksip-back-end** en Azure App Service.

3. **Aprovisionamiento de recursos en Azure:** Azure gestiona automáticamente la creación de los recursos necesarios en la nube (servidor, red, almacenamiento), proceso que tarda entre 2 y 5 minutos. Una vez completado, el servicio queda activo y accesible mediante la URL generada.

4. **Configuración de variables de entorno:** Tras el despliegue, se configuran las variables de entorno necesarias (como la cadena de conexión a la base de datos) desde la sección **Configuration** del servicio en el portal de Azure, sin necesidad de modificar el código fuente.

5. **Verificación del despliegue:** Una vez levantado el servicio, se verifica el correcto funcionamiento del backend accediendo a la URL generada con el sufijo `/swagger/index.html`, donde Swagger UI expone todos los endpoints disponibles para su validación manual.


**Componentes del Pipeline del Frontend (Vercel para Vue + Vite y Landing Page):**

1. **Detección de cambios en el repositorio:** Al realizar un commit en la rama principal de los repositorios **stocksip-front-end-application** o **stocksip-landing-page** en la organización **LiquoTrack-EXP** de GitHub, Vercel detecta automáticamente el cambio e inicia el pipeline de construcción y despliegue.

2. **Compilación del frontend:** Vercel ejecuta el proceso de construcción según el preset configurado: **Vite** para la aplicación web frontend desarrollada en Vue, y **Other** para la landing page estática en HTML, CSS y JavaScript. Durante este proceso, Vercel inyecta las variables de entorno definidas (como `VITE_API_BASE_URL`) en el bundle de producción.

3. **Despliegue en producción:** Si el proceso de construcción finaliza sin errores, Vercel publica automáticamente la nueva versión en producción. La landing page queda accesible bajo el dominio **stocksip-landing-page.vercel.app** y la aplicación web bajo **stocksip-front-end-application-navy.vercel.app**, ambas distribuidas globalmente a través de la CDN de Vercel.

4. **Vista previa por rama (Preview Deployments):** Vercel genera automáticamente una URL de vista previa por cada rama o pull request activo, permitiendo al equipo revisar los cambios antes de fusionarlos a la rama principal y que lleguen a producción.

5. **Invalidación de caché:** Vercel invalida automáticamente la caché tras cada nuevo despliegue, garantizando que los usuarios siempre reciban la versión más reciente de la aplicación sin necesidad de limpiar manualmente el caché del navegador.


**Componentes del Pipeline de las Aplicaciones Móviles (Firebase App Distribution y GitHub Releases):**

1. **Generación del APK firmado:** Desde **Android Studio**, se genera el APK firmado mediante la opción **Build → Generate Signed App Bundle or APK**, ingresando el key store path correspondiente. Este proceso produce el archivo binario listo para su distribución.

2. **Distribución interna con Firebase App Distribution:** El APK generado se sube a **Firebase App Distribution**, donde el equipo puede invitar a testers para que descarguen e instalen la aplicación directamente en sus dispositivos Android, permitiendo validar el funcionamiento antes de una distribución pública.

3. **Publicación en GitHub Releases:** El mismo APK se publica en un repositorio privado de GitHub bajo la sección **Releases**, creando una nueva versión etiquetada (por ejemplo, `v2.0.0`) con el archivo adjunto. Este proceso sigue la convención de **Semantic Versioning** definida por el equipo.

4. **Vinculación desde la Landing Page:** La URL de descarga del release en GitHub se asigna al botón de descarga de la landing page, permitiendo que los visitantes accedan directamente a la versión más estable de la aplicación móvil tanto para **Android nativo (Kotlin)** como para la **aplicación multiplataforma en Flutter**.
