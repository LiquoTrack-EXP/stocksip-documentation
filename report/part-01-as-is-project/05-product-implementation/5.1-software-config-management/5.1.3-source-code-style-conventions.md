### 5.1.3. Source Code Style Guide & Conventions

En esta sección, se definen las convenciones y referencias que se usaron para adoptar estrategias de nomenclatura de elementos de programación en los lenguajes que se usarán para la solución (HTML, CSS, JavaScript, C#, Vue y Base de Datos). En general, la nomenclatura de los archivos y secciones en la programación se hará en inglés.

* **Convenciones para HTML:** Para la codificación del proyecto en HTML, se utilizará el artículo “HTML Style Guide and Coding Conventions”. A continuación, se detallan las principales convenciones:
  * Los nombres de los elementos y atributos deben ser escritos en **minúsculas**.
  * Todas las etiquetas deben cerrarse correctamente.
  * Los valores de atributos deben ir entre comillas.
  * Utilizar metadatos esenciales como `charset` y `name="viewport"` para diseño responsivo.
  * Los nombres de archivos deben estar en minúsculas.

* **Nomenclatura en CSS:** Basado en “Google HTML/CSS Style Guide”:
  * Los estilos deben estar centralizados en archivos dedicados (por ejemplo, `styles.css`).
  * Usar clases en lugar de IDs para facilitar la reutilización.
  * Nombrar clases en minúsculas y separadas por guiones. Ejemplo: `nav-bar`.

* **Nomenclatura en JavaScript:** Basado en “Google JavaScript Style Guide”:
  * Evitar el uso de `var`, priorizando `let` y `const`.
  * Usar convención K&R para llaves.
  * Limitar líneas a 80 caracteres cuando sea posible.

* **Nomenclatura en C#:**
  * Clases, métodos y propiedades en **PascalCase**.
  * Variables locales y parámetros en **camelCase**.
  * Interfaces comienzan con `I`. Ejemplo: `IProductRepository`.
  * Enums en PascalCase, preferiblemente en plural. Ejemplo: `EProductStatuses`.

* **Nomenclatura en .NET Core:**
  * Campos privados en `_camelCase`.
  * Métodos asíncronos deben terminar en `Async`.
  * Métodos que llaman a funciones async también deben ser async.

* **Nomenclatura para RESTful API:**
  * Las URIs deben representar recursos usando sustantivos. Ejemplo: `api/products`.
  * Las acciones se definen mediante métodos HTTP (GET, POST, etc.).
  * Las rutas deben reflejar jerarquías de recursos.
  * Usar minúsculas y separar palabras con guiones.
  * Utilizar parámetros de consulta para filtrado. Ejemplo: `api/products?category=wine`.

* **Convenciones para MongoDB:**
  * No usar el prefijo `system` en colecciones.
  * Cada documento debe tener un `_id` único.
  * No duplicar campos dentro de un documento.
  * Usar camelCase para nombres de campos. Ejemplo: `orderDate`.

* **Nomenclatura para Kotlin:**
  * Paquetes en minúsculas sin guiones bajos.
  * Clases en **UpperCamelCase**.
  * Variables y funciones en **camelCase**.
  * Priorizar `val` sobre `var` para inmutabilidad.

* **Nomenclatura para Vue:**
  * Componentes en **PascalCase**. Ejemplo: `ProductCard.vue`.
  * Variables y funciones en **camelCase**.
  * Props en camelCase y uso en templates en **kebab-case**.
  * Eventos personalizados en kebab-case. Ejemplo: `update-stock`.
  * Composables con prefijo `use`. Ejemplo: `useInventory`.
  * Stores (Pinia) por contexto funcional. Ejemplo: `useInventoryStore`.
  * Rutas organizadas por módulos. Ejemplo: `/inventory`, `/reports`.
  * Templates limpios, modulares y reutilizables.
  * Estilos con clases descriptivas y consistentes.

Estas convenciones permiten mantener un código limpio, consistente y escalable en todo el proyecto.
