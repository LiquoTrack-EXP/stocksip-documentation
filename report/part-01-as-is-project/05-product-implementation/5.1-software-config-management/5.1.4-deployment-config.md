### 5.1.4. Software Deployment Configuration

En esta sección se describen únicamente las **configuraciones de despliegue** definidas para cada producto de software del proyecto. No se detallan pasos operativos ni procedimientos de publicación, ya que el objetivo de esta sección es evidenciar la infraestructura, servicios y parámetros utilizados para habilitar cada entorno de ejecución.

**Landing Page:**

Para el despliegue de la landing page se utilizó Vercel, una plataforma de alojamiento que permite publicar sitios estáticos directamente desde un repositorio de GitHub. Se importó el repositorio **stocksip-landing-page** desde la organización **LiquoTrack-EXP**, se verificó que el preset fuera **Other** (acorde a la naturaleza estática del sitio) y se seleccionó la carpeta `src` como directorio raíz. Una vez completada la configuración, se ejecutó el despliegue y la landing page quedó accesible bajo el dominio **stocksip-landing-page.vercel.app**, mostrando correctamente el hero section, las métricas clave y la sección de Misión, Visión y Valores.

**Aplicación Frontend Web:**

Para el despliegue de la aplicación web frontend también se utilizó Vercel. Se importó el repositorio **stocksip-front-end-application** desde la organización **LiquoTrack-EXP** en GitHub, se configuró el preset **Vite** correspondiente al framework utilizado y se añadió la variable de entorno **VITE_API_BASE_URL** apuntando a la URL base del backend. Tras ejecutar el despliegue, Vercel completó el proceso en aproximadamente 12 segundos y la aplicación quedó disponible en **stocksip-front-end-application-navy.vercel.app**, mostrando correctamente la pantalla de inicio de sesión con las opciones de ingreso por email, Google y GitHub.

**Base de Datos NoSQL:**

Para el despliegue de la base de datos se utilizó Azure Cosmos DB. Se inició sesión en el portal de Azure, se creó un nuevo recurso seleccionando **Azure Cosmos DB for MongoDB** con el tipo de clúster **vCore**, y se completaron los parámetros de configuración (nombre de cuenta, grupo de recursos, ubicación y opciones de rendimiento). Una vez creado el recurso, se habilitaron las reglas de firewall para permitir el acceso local desde MongoDB Compass a todos los miembros del equipo. Finalmente, se estableció la conexión utilizando la cadena de conexión proporcionada por Azure.

**Aplicación Backend:**

Para el despliegue del backend se utilizó Azure Web App. Se creó el repositorio en GitHub, se configuró el proyecto en Rider instalando las dependencias necesarias y se descargó el plugin **Azure Toolkit for Rider**. Una vez autenticada la cuenta de Azure desde el IDE, se publicó el proyecto seleccionando **Web App** como destino y asignando el dominio **stocksip-back-end**. La creación de los recursos en Azure tomó entre 2 y 5 minutos. Finalizado el proceso, se verificó el funcionamiento accediendo a la URL generada con el sufijo `/swagger/index.html`, y se configuraron las variables de entorno necesarias desde la sección **Configuration** del servicio en Azure.

**Aplicación Móvil en Android (Firebase App Distribution):**

Para el despliegue de la aplicación móvil en Android se utilizó Firebase App Distribution. Se creó el repositorio en GitHub y el proyecto en Android Studio con las dependencias necesarias. Luego, se creó un proyecto en Firebase, se registró la aplicación Android ingresando el nombre del paquete definido en `build.gradle.kt`, y se descargó el archivo `google-services.json` para colocarlo en la carpeta `app` del proyecto. Posteriormente, se integró el SDK de Firebase, se accedió a Firebase App Distribution y se generó el APK firmado desde **Build → Generate Signed App Bundle or APK**. Finalmente, el APK resultante fue subido a Firebase App Distribution para su distribución al equipo.
