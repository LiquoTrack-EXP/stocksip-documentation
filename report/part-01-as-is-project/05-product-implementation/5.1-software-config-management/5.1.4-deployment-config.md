### 5.1.4. Software Deployment Configuration

En esta sección se describen únicamente las **configuraciones de despliegue** definidas para cada producto de software del proyecto. No se detallan pasos operativos ni procedimientos de publicación, ya que el objetivo de esta sección es evidenciar la infraestructura, servicios y parámetros utilizados para habilitar cada entorno de ejecución.

#### Landing Page

La Landing Page se configura para su publicación mediante **GitHub Pages**, utilizando el repositorio correspondiente como fuente principal del contenido estático. La rama destinada al despliegue contiene los archivos finales del sitio, incluyendo HTML, CSS, JavaScript e imágenes.

La configuración considera los siguientes elementos:

- Repositorio público alojado en GitHub.
- Rama principal definida como fuente de publicación.
- Contenido estático preparado para distribución web.
- Acceso mediante enlace público generado por GitHub Pages.

#### Aplicación Web

La aplicación web se configura para su despliegue como una aplicación frontend desarrollada en **Vue**, utilizando un entorno de construcción optimizado para producción y consumo de servicios backend.

La configuración considera los siguientes elementos:

- Proyecto desarrollado en Vue con gestión de dependencias mediante Node.js.
- Proceso de build para generar archivos optimizados (HTML, CSS y JavaScript).
- Integración con servicios backend mediante endpoints REST.
- Uso de variables de entorno para la configuración de URLs y servicios.
- Estructura modular basada en componentes reutilizables.
- Preparación para despliegue en servicios de hosting web compatibles.

#### Base de Datos NoSQL

La base de datos NoSQL se configura mediante **Microsoft Azure Cosmos DB**, utilizando compatibilidad con **MongoDB** para facilitar la gestión de colecciones y documentos desde herramientas externas como MongoDB Compass.

La configuración considera los siguientes elementos:

- Creación de una cuenta de Azure Cosmos DB para MongoDB.
- Selección del tipo de despliegue compatible con aplicaciones NoSQL.
- Definición del grupo de recursos y la región de alojamiento.
- Habilitación de reglas de acceso para permitir conexiones autorizadas.
- Uso de una cadena de conexión para la administración de la base de datos desde entornos locales o remotos.

#### Aplicación Backend

La aplicación Backend se configura para su despliegue en **Microsoft Azure Web App**, con el fin de exponer los servicios del sistema a través de una URL pública.

La configuración considera los siguientes elementos:

- Repositorio independiente para el backend.
- Publicación bajo un servicio de tipo Web App.
- Definición del nombre del dominio de la aplicación.
- Activación de variables de entorno necesarias para el funcionamiento del sistema.
- Integración con Swagger para documentación y prueba de endpoints.

#### Aplicación Móvil Android Nativa

La aplicación móvil nativa en Android se configura para distribución mediante **Firebase App Distribution** y respaldo de versiones mediante **GitHub Releases**.

La configuración considera los siguientes elementos:

- Proyecto Android registrado en Firebase.
- Archivo `google-services.json` incorporado en el proyecto.
- Integración del SDK de Firebase en la aplicación.
- Generación de artefactos de distribución en formato APK.
- Uso de Firebase App Distribution para pruebas internas.
- Publicación de versiones estables mediante GitHub Releases.

#### Aplicación Multiplataforma

La aplicación multiplataforma se configura para su distribución mediante **GitHub Releases**, empleando un repositorio privado para almacenar las versiones estables del aplicativo.

La configuración considera los siguientes elementos:

- Repositorio privado destinado a la aplicación multiplataforma.
- Generación de artefactos instalables para Android.
- Publicación de versiones estables en GitHub Releases.
- Uso de la URL de descarga para vincular la distribución desde la plataforma principal.

#### Consideraciones Generales

Las configuraciones de despliegue establecidas permiten mantener separados los entornos de desarrollo, prueba y publicación de cada producto de software. De esta manera, se facilita la administración de las versiones, la distribución de los componentes y la disponibilidad de cada solución en su respectivo entorno.
